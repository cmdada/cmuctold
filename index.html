Evaluating the 2D Vector and Raster Graphics Processing Proficiency of CMU CS Academy's IDE

Introduction

This paper sets out to evaluate the computational capabilities of CMU Computer Science (CS) Academy's Integrated Development Environment (IDE) in processing 2D vector and raster graphics.

Methodology

The evaluation framework centers on the continuous creation and rendering of 2D objects while closely observing the total count of objects effectively processed by the CS Academy's IDE. The selection of test objects comprised fundamental shapes typical of vector imagery, along with raster text, found to fetch fonts from fonts.google.com based on network requests and debugging observations.

Evaluation Method

Performance testing was carried out using a system equipped with a 12th generation Intel Core i7 processor, 16GB RAM, and an Nvidia T1000 boasting 4GB of GDDR6, running on Windows 10. The testing spans were deployed on two prominent browsers: Firefox and Google Chrome.

Firefox demonstrated a robust performance with an average count of 5,423 shapes over ten trials, with each bout lasting near five minutes before the page became unresponsive, prompting either the browser or the system to crash. Conversely, Chrome illustrated a deal of inconsistency, as the quantities of shapes produced in each trial varied noticeably.

Such variability among the data emphasises that this test may not be optimal for Chromium-based browsers. It's worth noting that the performance log was recorded via Firefox's built-in profiler, accessible here.

Analysis of CMU CS Academy's Graphics Rendering

Take this with a grain of salt as this information has not been verified and is based on basic reverse engineering and speculation, but as far as I can tell, CMU CS Academy leverages Brython to execute code in a Python 3 environment, and Ace Editor as the code editor. While Brython supports outputting to a display, doing so requires specific JS frameworks that support doing so, and due to CMU's use of Ace, it seems that none of these frameworks would be compatible. To avoid this, they wrote a graphic library that renders directly to the webpage, but did not make it able to handle multiple shapes at once. To get around this, upon a shape or a text label being created, CMU CS Academy spawns a new process of their JS graphics library.
